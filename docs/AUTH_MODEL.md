# Auth Model (Supabase + Backend API)

## Responsibility split

- Supabase handles user login and session management.
- WaxWatch backend validates the JWT and authorizes all `/api/**` routes.

## Frontend requirements

- Obtain Supabase access token for the current session.
- Send it as:
  - `Authorization: Bearer <jwt>`

## Auth failures

- On 401/403:
  - clear local session via Supabase client
  - redirect to login route

Do not attempt to “fix” tokens client-side beyond refreshing via Supabase.

## Mobile handoff

React Native should open a secure web auth/handoff flow for sign-up/login/account/subscription management and then resume mobile via a validated deep-link callback.

### Web-only routes

These routes are owned by the web app and should be treated as the canonical place for auth/account/subscription UI:

- `/login`
- `/signup`
- `/account`
- `/account/subscription`

RN should navigate users to those routes in a browser/webview when auth/account/subscription actions are needed.

### Return URL / deep-link contract

The web auth routes accept these query params:

- `return_to` (optional): a web route to continue within web auth/account flows.
  - Must be a relative path and must match the allowlist above.
- `handoff` (optional): mobile callback URL.
  - Allowed forms: `waxwatch://...`, `waxwatch-dev://...`, or `https://*.waxwatch.app/...`.
- `state` (required when `handoff` is set): CSRF binding value generated by RN/backend.
- `nonce` (required when `handoff` is set): one-time replay-protection value.
- `expires_at` (required when `handoff` is set): expiry timestamp (ISO-8601 or UNIX seconds).

Recommended callback format back to RN:

- `waxwatch://auth/callback?status=<success|cancel|error>&state=<state>&code=<code>&error_code=<error_code>&error_message=<error_message>`

The callback should echo `state`, include either an authorization `code` (success) or a machine-readable `error_code`/`error_message`, and omit unused fields.

### Outcomes and security checks

Expected outcomes for web-to-mobile handoff:

- `success`: user authenticated and (if needed) completed account/subscription updates in web.
- `cancel`: user exited flow without completing action.
- `error`: flow failed (validation failure, expired request, provider failure, etc.).

Security checks before redirecting to mobile:

- Validate `state` matches server-side/session-bound value.
- Validate `nonce` is present, unconsumed, and bound to the same request.
- Validate `expires_at` is not expired.
- Reject untrusted `handoff` origins/schemes and non-allowlisted `return_to` values.
- Perform one-time use invalidation after success/cancel/error callback is generated.

### Backend/frontend requirements for session continuation

To keep subscription purchase flows out of RN UI, web remains the source of truth for account + billing actions.

Backend requirements:

- Provide a token exchange endpoint or session continuation endpoint for RN after web handoff success.
  - Example: exchange short-lived `code` for mobile session tokens.
- Bind exchange artifacts (`code`, `state`, `nonce`) to user + device context and enforce short TTL.
- Mark exchange artifacts one-time-use and revoke on replay attempts.
- Return subscription/account entitlements from backend APIs so RN can render state without in-app purchase logic.

Frontend requirements:

- RN launches web auth/account/subscription routes and never embeds native in-app purchase handling for these flows.
- RN handles callback `status` and exchanges `code` with backend to resume session.
- RN stores resulting session credentials securely and uses shared API auth adapter for authenticated calls.
- Web should handle billing/account UX end-to-end and only hand back the continuation token/code + status.
